### TV/라이브커머스 (공개 API 없음, MD 계약 기반)

CJ온스타일, GS샵, 롯데홈쇼핑, 현대홈쇼핑 4개 몰 모두 **공개 셀프서비스 오픈API가 없다** — 확인된 공통 패턴:

- 입점은 MD 심사/협상 기반 (자체 등록 불가)
- 승인 후에만 로그인 게이트된 파트너시스템(Withnet, 파트너지원시스템, hmall 파트너시스템 등) 접근 가능 — 상품등록 필드 스키마는 이 안에만 존재, 공개 문서 없음
- 계정 레벨 요건은 일부 공개됨: 보증보험(TV 1천만원/인터넷 5백만원 — 롯데홈쇼핑 기준), 전자계약(공인/공동인증서), PL보험, 원산지증명서·KC성적서·수입신고필증 등 서류 제출

→ 이 4개 몰은 "쇼핑몰별 필드" 실제 입력폼보다는 **"담당 MD 연락처/협력사코드/전자계약 여부" 같은 계정 메타데이터** 정도로 다루는 게 현실적. 상품등록 필드 자체는 파트너 계약 후 별도 확인 필요.

### 자료 부족 (공개 문서 거의 없음)

**하프클럽**: 검색/공식 페이지/연동사(카페24, 샵링커) 어디서도 필드 수준 문서를 찾지 못함. 로그인 게이트된 판매자시스템(seller-club.co.kr, scm.halfclub.com)에만 존재하는 것으로 추정.

---

**중요한 설계 시사점**: 원산지/인증정보/부가세유형/성인상품여부는 확인된 대다수 몰에 공통으로 존재한다 — 몰마다 고유한 게 아니라 **한국 전자상거래법상 공통 요구사항**이고, 몰마다 API 필드명/스키마만 다르다. "쇼핑몰별 필드" 영역에 몰마다 따로 넣기보다, **공통 필드 영역에 추가하고 저장 시 몰별 API 스키마로 매핑**하는 방향을 다음 브레인스토밍에서 검토할 것.

진짜 "쇼핑몰별 필드"로 남겨야 할 항목은 위 비교표의 "몰 특유 항목" 행 및 아래 "몰별 상세" 절 참고.

쿠팡/지마켓/인터파크/SSG닷컴의 반품지·출고지가 "사전 등록된 엔티티를 코드로 참조"하는 방식인 점은, 현재 `MallAddress` 타입(`code`/`name`/`zipCode`/`address`/`addressDetail`)이 이미 `code` 필드를 갖고 있어 방향이 맞아떨어진다.

공식 소스: [developers.coupangcorp.com](https://developers.coupangcorp.com) Open API 문서, Coupang Marketplace 셀러 유니버시티

|---|---|---|
| `returnCenterCode` (반품지 코드) | 필수 | 반품센터 식별자. 사전 등록 없으면 `"NO_RETURN_CENTERCODE"` + 인라인 주소 |
| `outboundShippingPlaceCode` + `remoteInfoId` (출고지 코드) | 필수 | 별도 "출고지 조회" API로 얻는 식별자 |
| `noticeCategories` / `noticeCategoryDetailName` (상품고시정보) | 필수(카테고리별) | 전자상거래법 고시 항목. 카테고리 메타데이터 API로 `MANDATORY` 여부 확인 가능 |
| `certificationType` / KC 인증번호 | 필수(전자제품/유아용품/조명/배터리 등) | KATS DB와 실시간 검증 |
| `requiredDocuments` | 필수(제한 카테고리) | 수입신고필증 등 서류 첨부 |
| `adultOnly` (성인상품 여부) | 선택 | 19+ 플래그 |
| `parallelImported` (병행수입 여부) | 필수(enum) | `NOT_PARALLEL_IMPORTED` 등 |
| `overseasPurchased` (해외구매대행 여부) | 필수(enum) | KC 예외 판정에 영향 |
| `afterServiceInformation` / `afterServiceContactNumber` | 필수 | A/S 연락처 |
| `taxType` | - | Sales Detail Query 응답에서 확인, 정확한 enum은 미확인 |
